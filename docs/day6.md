## 💡 Kernel 4核心思想

### 三句话总结
```
1. 每个线程不再只计算1个C元素，而是计算8个（TM=8）
2. 把B的值缓存在寄存器tmpB中，读1次用8次
3. 这样减少了Shared Memory访问，提高了数据复用率
```

---

### 与Kernel 3的对比

| 特性 | Kernel 3 | Kernel 4 |
|------|---------|---------|
| 每线程计算 | 1个C元素 | 8个C元素（TM=8） |
| 线程数 | 1024 | 512 |
| Block大小 | 32×32 | 64×64 |
| B值复用 | 每次从Shared Mem读 | 缓存在寄存器tmpB |
| 数据流 | Shared Mem → 计算 | Shared Mem → 寄存器 → 复用8次 |
| 性能 | 1281 GFLOPS | 3825 GFLOPS |

## 🔍 核心优化点

### 1. 寄存器复用（最重要）
```cpp
for (uint dotIdx = 0; dotIdx < BK; ++dotIdx) {
  float tmpB = Bs[dotIdx * BN + threadCol];  // 读1次，存寄存器
  
  for (uint resIdx = 0; resIdx < TM; ++resIdx) {
    threadResults[resIdx] += As[...] * tmpB;  // tmpB用8次
  }
}
```

**效果**：
- Kernel 3：每次计算都从Shared Memory读B
- Kernel 4：B读1次，在寄存器中用8次
- 减少Shared Memory访问 → 性能提升

---

### 2. 增加每线程工作量
```
Kernel 3:
  1024个线程，每个算1个C元素
  → 每个线程工作量小，寄存器利用率低

Kernel 4:
  512个线程，每个算8个C元素
  → 每个线程工作量大，数据在寄存器中复用
  → 指令级并行性更好
```

---

### 3. blockIdx映射优化
```cpp
// Kernel 4的配置（快30%）
const uint cRow = blockIdx.y;  // y → 行
const uint cCol = blockIdx.x;  // x → 列

// Host端
dim3 grid(N/BN, M/BM);  // (列数, 行数)
```

**为什么这样快？**

- GPU倾向先调度blockIdx.x连续的blocks
- 这些blocks执行时间接近，可能共享L2 cache
- 让它们访问A的同一行（连续内存）
- 比访问B的同一列（跳跃内存）效率高
- 结果：L2 cache命中率提升 → 性能+30%

## 📖 重要概念理解

### 1. BM、BN、BK、TM的含义
```cpp
template <const int BM, const int BN, const int BK, const int TM>

BM = 64  // Block处理C的行数（不是线程数！）
BN = 64  // Block处理C的列数
BK = 8   // Tile在K方向的大小
TM = 8   // 每个Thread计算的C元素个数

线程数 = (BM / TM) × BN = (64 / 8) × 64 = 512
```

**关键**：BM/BN/BK是矩阵块的大小，不是线程数！

---

### 2. 指针预移动的直觉理解
```cpp
A += cRow * BM * K;
```

**多种类比**：

**停车场类比**（最直观）：
```
停车场有M层，每层K个车位
要去第cRow个块（BM层）的起始

前面有多少层？cRow × BM层
每层K个车位
总共要跳过：cRow × BM × K个车位

所以：A += cRow * BM * K
```

**本质**：
```
跳过的元素数 = 跳过的行数 × 每行元素数
              = (cRow × BM) × K
              = cRow × BM × K
```

---

### 3. blockIdx映射和Host配置的关系

**重要认知**：blockIdx.x/y只是标签，含义由Kernel映射+Host配置共同决定

**配置1（Kernel 3风格）**：
```cpp
// Kernel
cRow = blockIdx.x;  // x → 行
cCol = blockIdx.y;  // y → 列

// Host
dim3 grid(M/32, N/32);  // (行数, 列数)

// 第一行blocks
Block(0,0), Block(0,1), Block(0,2), Block(0,3)
```

**配置2（Kernel 4风格）**：
```cpp
// Kernel
cRow = blockIdx.y;  // y → 行
cCol = blockIdx.x;  // x → 列

// Host（注意交换！）
dim3 grid(N/64, M/64);  // (列数, 行数)

// 第一行blocks（blockIdx.y=0）
Block(0,0), Block(1,0), Block(2,0), Block(3,0)
```

**关键**：
- blockIdx标号不同，但覆盖的C矩阵区域完全相同
- 两者配合才是完整系统
- 只要一致，怎么映射都对

---

## 🎯 三层理解框架

### 简化记忆（核心）
```
第1层：Grid层（粗定位）
  Block(cRow, cCol) → 负责C矩阵的哪个64×64块

第2层：Block层（细定位）
  线程(threadRow, threadCol) → 在块内的位置
  每个线程负责同一列的8个元素

第3层：Thread层（具体计算）
  寄存器：threadResults[TM] 存8个累加结果
  tmpB缓存B的值，读1次用8次

所有复杂的索引计算都是在实现这三层！
```

**答**：

```
1. GPU倾向先调度blockIdx.x连续的blocks
2. 这些blocks时间接近，可能共享L2 cache
3. 让它们访问A的同一行（连续内存）

```

---

### Q2: 指针预移动怎么具象化理解？

**答**：
```
类比停车场/书架/楼层：

前面有 cRow×BM 行（层/本）
每行有 K 个元素（车位/页）
总共跳过：cRow × BM × K 个元素

不需要记公式，理解"跳到自己的工作区"就够了
```

---

### Q3: 感觉要记的东西太多怎么办？

**答**：
```
不需要记所有细节！

只需要理解：
✓ 优化思想（寄存器复用、数据重用）
✓ 整体结构（Grid-Block-Thread三层）
✓ 为什么快（减少内存访问、增加并行）

具体的索引计算？
→ 理解原理比记公式重要100倍
→ 需要时查代码
→ 实践中会慢慢熟悉
```

---

### Q4: 更换索引后，C矩阵块排布变了吗？

**答**：
```
从blockIdx标号看：变了
  配置1: (0,0), (0,1), (0,2), (0,3)
  配置2: (0,0), (1,0), (2,0), (3,0)

从C矩阵覆盖区域看：没变
  两种配置第一行blocks都覆盖相同区域

为什么？
  Host配置也跟着变了
  Kernel映射 + Host配置 = 完整系统
```
